# Инвентарь

> **Status**: In Design
> **Author**: User + Claude
> **Last Updated**: 2026-04-15
> **Implements Pillar**: Pillar 1 (Рост героя), Pillar 5 (Понятная глубина)

## Overview

Inventory — система хранения предметов, которые игрок подбирает в данже. Хранит Equipment, Consumable и Junk; QuestItem живёт отдельно в `PlayerData.quest_items`. Ёмкость ограничена (20 слотов в рюкзаке). Игрок экипирует предметы в 8 слотов снаряжения (weapon, helmet, armor, gloves, boots, ring × 2, amulet), использует расходники через хоткей или UI, продаёт Junk и дублирующуюся экипировку в магазине. Инвентарь персистируется через SaveSystem и сбрасывается при старте новой игры. Основной интерфейс — SidePanel (Tab) во вкладке «Рюкзак».

## Player Fantasy

Открыть рюкзак после хорошего данжа — маленький ритуал. Смотришь что набрал, примеряешь: этот меч лучше текущего? Меняешь — и в следующей комнате чувствуешь разницу сразу. К концу данжа герой одет совсем иначе чем в начале — не просто сильнее в цифрах, а видно: другая броня, другое оружие. Инвентарь не должен ощущаться как задача («разберись с хламом») — только как витрина того, что уже заработано.

## Detailed Design

### Core Rules

**Рюкзак:**
1. Рюкзак вмещает 20 слотов; каждый слот занимает один предмет (Consumable стакается до 5 штук в один слот)
2. `add_item(item_id)` → если слотов нет → возвращает `false`; лут остаётся в мире (Loot System его не подбирает)
3. Предметы можно: экипировать, использовать (Consumable), выбросить (Equipment/Junk/Consumable), продать (в магазине)
4. QuestItem не хранится в рюкзаке — только в `PlayerData.quest_items`; не подчиняется правилам инвентаря

**Слоты экипировки (8):**
5. Слоты: weapon, helmet, armor, gloves, boots, ring_1, ring_2, amulet
6. Экипировка предмета: предмет из рюкзака → перемещается в слот; предыдущий предмет → возвращается в рюкзак
7. Если рюкзак полон при попытке снять предмет → снять нельзя (нет места для замены)
8. Бонусы экипированных предметов применяются к `StatsComponent` немедленно при экипировке и снимаются при снятии

**Потребление:**
9. Consumable используется из рюкзака или хоткея → эффект применяется к игроку → стак уменьшается на 1; при 0 — слот освобождается
10. Хоткеи расходников: слоты 1–4 — четыре выделенных слота, назначаются вручную из рюкзака

**Сброс:**
11. Содержимое рюкзака и экипировки сохраняется между визитами в данж
12. При смерти в данже — инвентарь сохраняется (предметы не теряются)
13. При старте новой игры (New Game) — инвентарь полностью сбрасывается

### States and Transitions

**Состояния слота рюкзака:**

| Состояние | Условие | Доступные действия |
|-----------|---------|-------------------|
| `EMPTY` | Слот пуст | — |
| `OCCUPIED` | Предмет лежит в слоте | Экипировать / Использовать / Выбросить / Продать |
| `STACKED` | Consumable, стак < 5 | Использовать / Выбросить (весь стак) |

**Состояния слота экипировки:**

| Состояние | Условие | Доступные действия |
|-----------|---------|-------------------|
| `EMPTY` | Ничего не надето | Экипировать подходящий предмет |
| `EQUIPPED` | Предмет надет | Снять (если есть место в рюкзаке) / Заменить |

**Переходы:**
```
EMPTY → OCCUPIED        : add_item() успешен
OCCUPIED → EMPTY        : выброс / продажа / использование последнего стака
OCCUPIED → EQUIPPED     : игрок экипирует предмет из рюкзака
EQUIPPED → OCCUPIED     : игрок снимает предмет (рюкзак не полон)
EQUIPPED → EQUIPPED     : замена — старый идёт в рюкзак, новый в слот
OCCUPIED → STACKED      : add_item() с тем же Consumable, стак < 5
STACKED → OCCUPIED      : стак = 1 (один оставшийся)
STACKED → EMPTY         : используется последний расходник
```

### Interactions with Other Systems

| Система | Направление | Интерфейс |
|---------|-------------|-----------|
| **Loot System** | upstream → | `Inventory.add_item(item_id)` → возвращает `bool`; `false` = полный |
| **Item Database** | upstream → | `ItemDatabase.get_item(item_id)` → данные предмета (тип, слот, бонусы) |
| **StatsComponent** | downstream → | `StatsComponent.apply_equipment_bonus(item_id)` / `remove_equipment_bonus(item_id)` при экипировке/снятии |
| **Shop** | ↔ bidirectional | Магазин читает `Inventory.get_all()` для UI; вызывает `Inventory.remove_item()` при продаже и `Inventory.add_item()` при покупке |
| **Quest System** | downstream → | QuestItem НЕ хранится в Inventory; `PlayerData.quest_items[]` — отдельный массив |
| **SaveSystem** | ↔ bidirectional | `Inventory.serialize()` → сохраняет `backpack[]` и `equipped[]`; `Inventory.deserialize()` → восстанавливает |
| **UI Инвентаря** | downstream → | UI читает `Inventory.backpack[]` и `Inventory.equipped[]`; подписывается на сигнал `inventory_changed` |
| **HUD** | downstream → | Читает `Inventory.hotkey_slots[0..3]` для отображения слотов 1–4 и кол-ва расходников |

## Formulas

- **Свободные слоты**: `free_slots = BACKPACK_SIZE - count(occupied_slots)` где стак занимает 1 слот независимо от кол-ва
- **Стоимость продажи**: `sell_price = item.sell_value` (определяется в Item Database; Junk = 5–50, Equipment = 10–500 по редкости)
- **Результирующий стат от экипировки**: `effective_stat = base_stat + Σ(equipped_item.bonus[stat])` — вычисляется в StatsComponent
- **Ёмкость стака**: `max_stack = 5` для всех Consumable (задаётся в Item Database)

## Edge Cases

1. `add_item()` при полном рюкзаке → возвращает `false`; лут остаётся в мире со своим таймером
2. Экипировка предмета при полном рюкзаке → если в слоте уже что-то есть → замена невозможна (нет места для старого предмета); если слот пуст → экипировка без перемещения в рюкзак
3. Выброс в данже → предмет исчезает безвозвратно (не создаётся лут-нода); предупреждение в UI
4. Попытка продать QuestItem → `false`; кнопка «Продать» неактивна для QuestItem
5. Стак Consumable переполнен (уже 5/5) → `add_item()` ищет другой свободный слот; если нет — возвращает `false`
6. Использование расходника вне боя → эффект всё равно применяется; нет ограничения «только в бою»
7. Снятие экипировки при рюкзаке 20/20 → действие заблокировано; UI показывает «Нет места в рюкзаке»
8. Два кольца одного типа/редкости → разрешено (ring_1 и ring_2 независимы)

## Dependencies

| Система | Тип зависимости | Интерфейс |
|---------|----------------|-----------|
| **Loot System** | Hard — источник предметов | `add_item(item_id) → bool` |
| **Item Database** | Hard — без данных предмета нельзя ничего сделать | `get_item(item_id)` |
| **StatsComponent** | Hard — экипировка без применения бонусов бессмысленна | `apply_equipment_bonus()`, `remove_equipment_bonus()` |
| **SaveSystem** | Hard — персистентность | `serialize()`, `deserialize()` |
| **Shop** | Soft — продажа/покупка улучшает экономику | `remove_item()`, `add_item()`, `get_all()` |
| **UI Инвентаря** | Soft — без UI инвентарь недоступен игроку; в headless-тестах не нужен | сигнал `inventory_changed` |
| **Quest System** | Soft — QuestItem хранится отдельно; Inventory только знает «не трогать» | проверка `item.type == QUEST_ITEM` |

## Tuning Knobs

| Knob | Текущее | Безопасный диапазон | Эффект при изменении |
|------|---------|--------------------|--------------------|
| `BACKPACK_SIZE` | 20 | 12–30 | Ниже = больше давления выбора; выше = меньше решений |
| `MAX_STACK` | 5 | 3–10 | Кол-во зелий в одном слоте; влияет на выживаемость |
| `sell_value` (по редкости) | 10–500 | ×0.5–×2.0 | Скорость накопления золота |
| `HOTKEY_SLOTS` | 4 (клавиши 1–4) | 2–6 | Кол-во быстрых слотов расходников; больше = сложнее UI |

## Acceptance Criteria

1. Подбор предмета в данже → предмет появляется в рюкзаке; `free_slots` уменьшается на 1
2. Рюкзак полон (20/20) → лут не подбирается; предмет остаётся в мире
3. Экипировка оружия → старое оружие переходит в рюкзак; `StatsComponent` немедленно отражает новые статы в HUD
4. Использование расходника через хоткей 1–4 → эффект применяется; стак уменьшается на 1; при 0 — хоткей-слот пустой
5. Продажа Junk в магазине → предмет исчезает из рюкзака; золото зачислено
6. Попытка продать QuestItem → кнопка неактивна; предмет не исчезает
7. Сохранение и загрузка → рюкзак и экипировка восстановлены точно (типы, редкость, стаки)
8. Снятие предмета при рюкзаке 20/20 → действие заблокировано с пояснением в UI
