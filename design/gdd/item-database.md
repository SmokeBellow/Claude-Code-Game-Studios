# База данных предметов (Item Database)

> **Status**: Approved
> **Author**: User + Claude Code Game Studios
> **Last Updated**: 2026-03-24
> **Implements Pillar**: Рост героя (Pillar 1), Понятная глубина (Pillar 5)

## Overview

База данных предметов — статический реестр всех предметов в игре: их определений, характеристик, редкости и аффиксов. Система не управляет логикой лута или инвентаря — она только хранит данные и предоставляет их другим системам по запросу. Предметы делятся на четыре класса: **экипировка** (оружие, шлем, броня, перчатки, сапоги, аксессуар) — надевается и даёт постоянные бонусы к атрибутам/статам; **расходники** (зелья, свитки) — используются один раз и исчезают; **квестовые предметы** — необходимы для выполнения или продвижения квестов, нельзя продать или выбросить; **мусор** — не несёт боевой ценности, продаётся НПС за золото. Экипировка и расходники имеют один из четырёх уровней редкости (Common / Uncommon / Rare / Epic); квестовые предметы и мусор редкости не имеют. База данных является источником истины о предметах для систем Лута, Инвентаря, Магазина и Квестов.

## Player Fantasy

Найти новый предмет — это момент надежды и возбуждения. Игрок открывает инвентарь и видит: синее имя вместо серого, незнакомый аффикс, цифры которые больше текущих. Каждый предмет рассказывает историю одним предложением: "Клинок Проклятого Рыцаря — +15 Силы, +8% шанс крита". Редкий предмет ощущается как награда, заработанная за умелую игру. Мусор продаётся без сожаления — он часть экономики. Квестовый предмет вызывает любопытство: зачем он нужен, кому его отдать? База данных — это «алфавит» мира: из этих предметов складывается история героя.

## Detailed Design

### Core Rules

**Классы предметов**

1. Все предметы делятся на **4 класса**: `Equipment`, `Consumable`, `QuestItem`, `Junk`.
2. `Equipment` и `Consumable` имеют поле редкости (Common / Uncommon / Rare / Epic).
3. `QuestItem` и `Junk` — отдельные подклассы без поля редкости. `QuestItem` нельзя продать или выбросить. `Junk` продаётся НПС за золото, боевой ценности нет. MVP-расходники — только зелья (мгновенное или длительное действие); свитки используют тот же подкласс `ConsumableItem` и не имеют отдельной механики в MVP.

**Слоты экипировки**

4. Всего **8 слотов**: Оружие, Шлем, Броня, Перчатки, Сапоги, Кольцо 1, Кольцо 2, Амулет.
5. Кольца используют общий пул предметов типа `Ring` — одно кольцо может быть надето в оба слота (кроме одинаковых `item_id`).

**Структура данных**

6. Каждый предмет — `Resource` с полями: `item_id`, `display_name`, `description`, `icon`, `rarity`, `sell_value`, `required_level`.
7. `EquipmentItem` добавляет: `slot`, `weapon_base_damage` (только для оружия), `attribute_bonuses: Array[StatBonus]`, `stat_bonuses: Array[StatBonus]`.
8. `ConsumableItem` добавляет: `effect_type`, `effect_value`, `buff_stat` (один стат), `effect_duration`, `max_stack`.
9. `StatBonus` — минимальный struct: `stat: StatType`, `value: float`.
10. MVP-предметы имеют хардкодированные бонусы (статичные значения в `.tres`). Динамические аффиксы с именованными префиксами/суффиксами (Blade of X, Ring of Y) — архитектурно зарезервированы, реализуются в Vertical Slice.

**Диапазоны значений по редкости (гайд для ручного авторинга)**

| Атрибут (два, суммарно) | +2–4 | +5–8 | +9–13 | +14–18 |

### States and Transitions

| Состояние | Применимо к | Условие входа | Условие выхода |
|-----------|-------------|--------------|----------------|
| **В базе** | Все типы | Определён в .tres файле | Никогда (данные постоянны) |
| **В инвентаре** | Equipment, Consumable, QuestItem, Junk | Лут-система добавила экземпляр | Использован / продан / выброшен |
| **Экипировано** | Equipment | Игрок надел в слот | Снято / заменено |
| **Заблокировано** | QuestItem | Добавлен в инвентарь | Квест завершён *(Provisional: Система квестов #21)* |

*Сама База данных состояний не меняет — она только читается. Состояния экземпляров предметов управляются Инвентарём (#17).*

### Interactions with Other Systems

| Система | Направление | Данные |
|---------|-------------|--------|
| **Лут-система** | ← Item Database | Читает определения предметов по `item_id` для генерации дропа |
| **Инвентарь** | ← Item Database | Читает полные определения предметов для отображения и применения бонусов |
| **Health & Stats** | ← Item Database | Читает `attribute_bonuses` и `stat_bonuses` при надевании/снятии экипировки |
| **Магазин** | ← Item Database | Читает `sell_value`, `display_name`, `icon` для отображения товаров |
| **Система квестов** | ← Item Database | Читает `QuestItem` определения; проверяет наличие нужного `item_id` в инвентаре |
| **Система способностей** | ← Item Database | Читает предметы с бонусами к `ABILITY_POWER` и `MAX_MANA` |

*Item Database — пассивная система: не обращается ни к чему, только предоставляет данные по запросу. Все данные хранятся как `.tres` ресурсы Godot, загружаются через `ResourceLoader`.*

## Formulas

**Формул вычисления нет** — Item Database хранит готовые значения, не вычисляет их.

**Ссылки на формулы других систем:**

```
# Применение бонуса атрибута от экипировки (Health & Stats):
total_attribute = base_attribute + sum(equipment.attribute_bonuses where stat == attribute)
effective_value = effective(min(total_attribute, HARD_CAP))  # кап = 70

# Применение бонуса производного стата:
final_derived_stat = formula(effective_value) + sum(equipment.stat_bonuses where stat == derived)

# Итоговый урон с оружием:
final_damage = (weapon_base_damage + phys_damage_bonus) * crit_multiplier
```
→ Источник формул: `design/gdd/health-stats.md`

**Цена продажи (гайд для авторинга):**

| Тип | Цена продажи НПС |
|-----|-----------------|
| Junk | 1–20 золота |
| Equipment Common | 10–30 золота |
| Equipment Uncommon | 31–80 золота |
| Equipment Rare | 81–200 золота |
| Equipment Epic | 201–500 золота |

## Edge Cases

| Ситуация | Поведение |
|----------|-----------|
| `item_id` не найден в базе | Логируется ошибка; предмет не создаётся; лут-система получает `null` |
| Два одинаковых кольца в обоих слотах | Запрещено: Инвентарь проверяет `item_id` — один и тот же `item_id` не может занимать оба слота одновременно |
| Бонус атрибута + базовый атрибут превышает хард кап 70 | `min(total, 70)` применяется в Health & Stats — Item Database хранит сырое значение, кап не её ответственность |
| `QuestItem` пытается продать/выбросить | Операция заблокирована на уровне Инвентаря; Item Database не хранит логику блокировки |
| `required_level` выше уровня игрока | Предмет нельзя надеть; в инвентаре отображается с предупреждением. Лут может выпасть, но использование заблокировано |
| Consumable с `effect_duration = 0` | Мгновенное действие: HP/мана восстанавливаются немедленно, без баффа |
| Consumable с `max_stack` при полном стаке | Новый предмет занимает отдельный слот инвентаря; если инвентарь полон — предмет не подбирается *(Provisional: логика Инвентаря #17)* |
| Файл `.tres` предмета повреждён или отсутствует | `ResourceLoader` возвращает `null`; система логирует ошибку; предмет пропускается |

## Dependencies

**Входящие зависимости (от кого зависит эта система):**

Нет. Item Database — Foundation слой, не зависит ни от одной другой системы.

**Исходящие зависимости (кто зависит от этой системы):**

| Система | Что берёт | Тип зависимости |
|---------|-----------|-----------------|
| **Лут-система** (#12) | Определения предметов по `item_id` для генерации дропа | Жёсткая |
| **Инвентарь** (#17) | Полные определения предметов для отображения и применения бонусов | Жёсткая |
| **Health & Stats** (#1) | `attribute_bonuses` и `stat_bonuses` при надевании/снятии экипировки | Жёсткая |
| **Магазин** (#22) | `sell_value`, `display_name`, `icon` для отображения товаров | Жёсткая |
| **Система квестов** (#21) | `QuestItem` определения; проверяет наличие нужного `item_id` в инвентаре | Жёсткая |
| **Система способностей** (#10) | Предметы с бонусами к `ABILITY_POWER` и `MAX_MANA` | Жёсткая |

**Интерфейс доступа:**

```gdscript
# Единственный способ получить предмет из базы:
var item: ItemResource = load("res://assets/data/items/{item_id}.tres")
# или через ItemRegistry (autoload) — детали уточняются в дизайне Инвентаря #17
```

*Все зависимости однонаправленные: Item Database никогда не обращается к другим системам.*

## Tuning Knobs

| Параметр | Текущее значение | Безопасный диапазон | Что ломается при выходе |
|----------|-----------------|---------------------|------------------------|
| Количество предметов экипировки (MVP) | 35 | 20–60 | <20: ощущение повторяемости лута; >60: перегрузка авторинга без Инвентаря и Лута |
| Цена продажи: Common Equipment | 10–30 золота | 5–50 | <5: золото теряет ценность; >50: игрок слишком богат без баланса магазина (#22) |
| Цена продажи: Epic Equipment | 201–500 золота | 100–1000 | Диапазон уточняется при дизайне Магазина (#22) — здесь только верхняя граница |
| Цена продажи: Junk | 1–20 золота | 1–50 | >50: мусор становится выгоднее экипировки, ломает экономику |
| `max_stack` расходников (по умолчанию) | 5 | 1–20 | <1: невозможно; >20: Инвентарь теряет смысл как ограничение |
| `required_level` минимальный | 1 | 1–15 | >15 для Common: новые игроки не находят ничего надеваемого |
| `effect_duration` максимальный (расходник) | не задан (определяется побуждом) | 0–300 сек | >300: постоянный эффект перекрывает всю боевую систему |

*Диапазоны значений атрибутных бонусов по редкости — см. таблицу в Core Rules. Они являются гайдом для авторинга, не ограничением в коде.*

## Visual/Audio Requirements

Item Database — пассивная система данных, визуальных или аудио-компонентов не имеет. Требования к ассетам:

- Каждый предмет должен иметь иконку: `@export var icon: Texture2D` — ссылка на `.png` ресурс Godot (не путь-строка)
- Формат иконок: 64×64 px, PNG с прозрачным фоном
- Именование: `icon_{item_id}.png` в `assets/art/ui/items/`
- Цветовая кодировка редкости применяется на уровне UI (Инвентарь #17), а не в данных предмета

*Звуковые эффекты при подборе предмета (лут-звук) — ответственность Лут-системы (#12), не Item Database.*

## UI Requirements

Item Database не владеет UI — она только предоставляет данные. Требования к данным для UI-систем:

| UI-система | Поля, которые она читает |
|------------|--------------------------|
| **Инвентарь** (#17) | `display_name`, `description`, `icon`, `rarity`, `required_level`, `sell_value`, `slot`, `attribute_bonuses`, `stat_bonuses` |
| **Магазин** (#22) | `display_name`, `icon`, `rarity`, `sell_value` |
| **Тултип предмета** | Все поля — отображает полную карточку предмета |
| **Система квестов / UI** (#28) | `display_name`, `icon` для отображения в журнале квестов |

*Визуальное оформление редкости (цвет имени, рамка иконки) — дизайн Инвентаря #17.*

## Acceptance Criteria

| # | Критерий | Как проверить |
|---|----------|---------------|
| 1 | `ItemResource.tres` загружается без ошибок через `ResourceLoader` | Запустить игру, проверить Output — нет ошибок парсинга ресурсов |
| 2 | `EquipmentItem` содержит корректный `slot` и непустой `item_id` | Открыть любой `.tres` экипировки в редакторе Godot, проверить поля |
| 3 | `ConsumableItem` с `effect_duration = 0` применяется мгновенно | Использовать зелье — HP/мана восстанавливаются сразу, без активного баффа |
| 4 | Несуществующий `item_id` возвращает `null` и логирует ошибку | Запросить `load("res://assets/data/items/invalid_id.tres")` — ошибка в логе, нет краша |
| 5 | `QuestItem` нельзя продать или выбросить | Попытка продать квестовый предмет в Магазине — операция заблокирована *(зависит от Инвентаря #17)* |
| 6 | Одинаковый `item_id` не может занять оба слота кольца | Попытаться надеть одно кольцо дважды — второй слот отказывает *(зависит от Инвентаря #17)* |
| 7 | MVP: 35 предметов экипировки определены в `.tres` файлах | Подсчитать `.tres` файлы в `assets/data/items/equipment/` — минимум 35 |
| 8 | Все предметы MVP имеют уникальный `item_id` | Скрипт-валидатор: загрузить все `.tres`, проверить уникальность `item_id` |
| 9 | `attribute_bonuses` и `stat_bonuses` экипировки корректно читаются Health & Stats | Надеть предмет с `+5 STR` — атрибут игрока увеличивается на 5 |
| 10 | Значения бонусов не превышают верхние границы гайд-таблицы | Ревью авторинга: проверить все Epic предметы — бонусы ≤ максимумов таблицы |
